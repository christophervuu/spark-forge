# PACKET

## Title

Short descriptive name for the change.

---

## Status

proposed | promote

proposed = design under consideration  
promote = update to durable specs required

---

## Goal

Describe the outcome this change aims to achieve.

Focus on the user or system impact.

---

## Scope

### In Scope

List the behaviors or components this packet will change.

### Out of Scope

Explicitly list what this packet will NOT change.

---

## Behavioral Changes

Describe the observable behavior differences before and after this change.

If applicable, include:

Before:
- previous behavior

After:
- new behavior

---

## Acceptance Examples

Provide concrete scenarios that demonstrate correct behavior.

Prefer deterministic examples such as:

Given / When / Then

or

Input → Output

Include edge cases when relevant.

---

## Implementation Outline

High-level approach for implementing the change.

This may include:

- affected subsystems
- new modules or files
- migration considerations
- risks or constraints

Do NOT include full implementations.

---

## Task Candidates

List candidate implementation tasks.

Tasks should be:

- atomic
- independently executable
- implementation-ready

Example:

- Add validation for X
- Update API endpoint Y
- Introduce module Z
- Add tests for new behavior

---

## Notes (Optional)

Any additional context that may help implementation agents.
